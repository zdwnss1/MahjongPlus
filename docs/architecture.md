# Architecture

## Non-negotiable principles

### Reality floor

The online table may do more than a physical table, never less. A player does not receive a whitelist of the only actions they are allowed to perform. Instead, the client may submit any `ActionAttempt` from the catalog at any revision. Suggested actions are ergonomic hints only.

The authoritative pipeline is:

```text
ActionAttempt
  -> stale / physical-reference checks
  -> base-rule violations
  -> ordered rule-module adjudication
  -> reject or commit
  -> penalty and rule effects
  -> ActionReceipt + event journal + new revision
```

A network-stale action is not a foul. A duplicate `attemptId` returns the original receipt and cannot apply its penalty twice.

### Fine-grained modules

The server now separates:

- `kernel/random.ts` — deterministic named random streams.
- `kernel/tileSet.ts` — arbitrary physical tile multiplicity and traits.
- `kernel/seededShan.ts` — deterministic wall adapter.
- `kernel/tileLedger.ts` — stable physical tile-instance identities and locations.
- `kernel/catalog.ts` — the always-available action vocabulary.
- `kernel/adjudicator.ts` — base adjudication and rule-module composition.
- `kernel/penalties.ts` — constitutional foul consequences.
- `kernel/ruleModules.ts` — executable rule-module boundary.
- `kernel/journal.ts` — append-only event identities and ordering.
- `gameHost.ts` — adapter between the modular kernel and the current standard-riichi engine.
- `room.ts` — room identity, constitution, governance, and match lifecycle only.

## Authority layers

1. **Safety/runtime kernel** — authentication, revisions, idempotency, projections, resource boundaries.
2. **Match constitution** — base profile, length, starting score, bankruptcy, rule slots, timeout, and default penalty policy.
3. **Physical table definition** — tile kinds, copy counts, red traits, dead wall, random streams.
4. **Base riichi profile** — versioned Tenhou-like or M.LEAGUE-like normative rules.
5. **Ordered player rule pack** — future executable MRIR modules.

Lower layers cannot be modified by higher layers unless an explicit capability is exported.

## Physical tiles

Wire state uses `TileInstanceView { id, face, physicalFace, traits }`. A discard refers to an instance ID, not merely `m5`. This allows future rules to distinguish two physically separate copies, attach traits, create fifth or sixth copies, and add extra red fives.

`TileSetDefinition` is independent from the standard engine. The current deterministic wall already accepts arbitrary copy counts. Some advanced shapes and calls still require the forthcoming local kernel fork because upstream Japanese-mahjong algorithms assume ordinary riichi semantics.

## Action hints versus action authority

`GameActionRequest` remains as an opportunity hint generated by the standard engine. It is not a permission boundary. The browser also displays the universal action panel.

All writes use:

```text
game:attempt {
  attemptId,
  observedRevision,
  action: ActionIntent
}
```

and return an `ActionReceipt` with outcome, violations, penalties, revisions, and committed event IDs.

## Current adapter boundary

The project still uses `@kobalab/majiang-core` for ordinary turn progression and scoring. `RuleAwareMajiangGame` injects a deterministic custom wall, while `GameHost` surrounds the engine with open action adjudication, physical identities, penalties, and receipts.

This is deliberately an adapter, not the final rule kernel. The next major change is a repository-local fork with hooks around:

- action opportunity generation;
- attempted-action validation;
- event commit;
- draw and wall mutation;
- hand interpretation;
- win evaluation;
- settlement;
- hand and match termination.

With no executable player rules, the local fork must remain differential-test equivalent to upstream.
